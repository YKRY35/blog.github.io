两种方式在Java层拿到Instrumentation：javaagent方式，attach方式

javaagent 方式

在应用启动前，打上javaagent参数。

java -javagent:/path/xxx.jar com.example.A

那么，xxx.jar中指定类(META-INF配置中指定)的premain方法就会在主类A的main方法执行前执行，并提供Instrumentation。

	}

应用照常启动。

java com.example.A

之后用别的进程侵入到目标应用进程。开一个新的Java进程，进程中：

VirtualMachine vmObj = null;
try {

sandbox类隔离设计

class SandboxClassLoader extends URLClassLoader {

    private final String toString;
    private final String path;

    SandboxClassLoader(final String namespace,
                       final String sandboxCoreJarFilePath) throws MalformedURLException {
        super(new URL[]{new URL("file:" + sandboxCoreJarFilePath)});
        this.path = sandboxCoreJarFilePath;
        this.toString = String.format("SandboxClassLoader[namespace=%s;path=%s;]", namespace, path);
    }

    @Override
    protected synchronized Class<?> loadClass(String name, boolean resolve) throws ClassNotFoundException {
        final Class<?> loadedClass = findLoadedClass(name);
        if (loadedClass != null) {
            return loadedClass;
        }
//        // 优先从parent（SystemClassLoader）里加载系统类，避免抛出ClassNotFoundException
//        if(name != null && (name.startsWith("sun.") || name.startsWith("java."))) {
//            return super.loadClass(name, resolve);
//        }
        try {
            Class<?> aClass = findClass(name);
            if (resolve) {
                resolveClass(aClass);
            }
            return aClass;
        } catch (Exception e) {
            return super.loadClass(name, resolve);
        }
    }

可以看到SandboxClassLoader遇到需要load的类，他会优先自己去findClass,defineClass，优先由自己加载，不干扰它的父加载器。

当自己加载不到时，再去委派给父加载器去loadClass。

类隔离原理

JVM何时触发类加载

JVM规范，似乎没有规定要什么时候加载类，只规定了何时应当初始化类，因此类加载时机取决于虚拟机实现。

下面以hotspot虚拟机为例(openjdk, oracle jdk)，介绍其类加载机制。

类是懒加载的。简单说，就是用到目标类时，发现目标类还未被当前加载器加载，则触发加载。

一般如下情况触发加载：

  1. 加载类A时，类A的父类和接口会被加载

  2. 类被new时，或者静态方法调用(invokestatic)、字段被访问时(getstatic,putstatic) 会加载类

  3. 反射

举例

public class AAA{
	public static void main(String[] args){
		java.lang.Float.valueOf("1.0f"); // 断点1
	}
}
public class ClassLoader{
	public Class<?> loadClass(String name) throws ClassNotFoundException {
        return loadClass(name, false); // 断点2
    }
}

先打断点1，然后启动应用，然后打断点2，继续执行。

会发现断点2处，name为 java/lang/Float，说明在执行main方法的第一句话时，Float这个类才被触发加载，在类AAA被加载时，Float类并不会被加载。

上面的类AAA，在被JVM加载链接初始化后，在内存里大致长这样。

Klass AAA是AAA类在内部的结构，里面包括方法（方法中给出了字节码指令），包括了常量池（最左列#是下标，省去了一部分，因此看着不连续;中间是类型，详见虚拟机规范；第三列是值，可能是常量池下标，可能是内存地址）。

SymbolTable是字符串常量池，对JVM来说是全局的。

此时，main方法未被执行，可以看到常量池的#21，Class类型的槽(slot)指向的是Symbol*，指向字符串常量池的字符串，还是处于符号引用。

当main方法执行时，执行到invokestatic，发现方法描述在类常量池的#3个槽（实际情况是，存在ConstantPoolCache，这里涉及到类链接时的指令重写rewrite，字节码会被重写成cache表下标，但这里为了方便解释类加载问题，暂时先忽略指令重写），这时，知道了类描述是#21个槽，而方法描述是在#22个槽。

首先要先获取类，看了看#21个槽，发现这个指针指向的是字符串Symbol*，而非类的内部表示Klass*，意识到类尚未解析，触发resolve_class解析类。

之后，这个类再次用到java/lang/Float，由于常量池已经保存了解析后的类地址，因此不会再触发解析和加载。

PS. 有同学可能会问，C++中指针就一地址，我怎么快速区分它到底是 符号引用Symbol*还是地址引用Klass*。看了这constantPool.hpp这段就明白了：

class CPSlot VALUE_OBJ_CLASS_SPEC {
  intptr_t _ptr;
 public:
  CPSlot(intptr_t ptr): _ptr(ptr) {}
  CPSlot(Klass* ptr): _ptr((intptr_t)ptr) {}
  CPSlot(Symbol* ptr): _ptr((intptr_t)ptr | 1) {}

  intptr_t value()   { return _ptr; }
  bool is_resolved()   { return (_ptr & 1) == 0; }
  bool is_unresolved() { return (_ptr & 1) == 1; }

  Symbol* get_symbol() {
    assert(is_unresolved(), "bad call");
    return (Symbol*)(_ptr & ~1);
  }
  Klass* get_klass() {
    assert(is_resolved(), "bad call");
    return (Klass*)_ptr;
  }
};

Symbol*地址保存时，最低位被置为1了，取时再还原，用以区分。（8字节为单位去寻址，地址低3位必是000）

JVM哪个加载器来加载

JVM在解析类时，发现类未加载，会使用定义当前类的类加载器(defining loader)去加载目标类。

defining loader: 定义它的加载器

initiating loader: 触发加载它的加载器

例如，还是上面那个例子。

public class AAA{
	public static void main(String[] args){
		java.lang.Float.valueOf("1.0f"); 
	}
}

打断点2的时候，会发现ClassLoader的this是AppClassLoader，而不是null (bootstrap classloader)。

因为，它用AAA的加载器AppClassLoader去尝试加载java/lang/Float了。

从constantPool.cpp(节选)源码中可以证明这一点：

Klass* ConstantPool::klass_at_impl(constantPoolHandle this_oop, int which, TRAPS) {
  Symbol* name = NULL;
  Handle       loader;
  {  MonitorLockerEx ml(this_oop->lock());

    if (this_oop->tag_at(which).is_unresolved_klass()) {
      if (this_oop->tag_at(which).is_unresolved_klass_in_error()) {
        in_error = true;
      } else {
        do_resolve = true;
        name   = this_oop->unresolved_klass_at(which); // 拿到目标类名
        loader = Handle(THREAD, this_oop->pool_holder()->class_loader()); //拿到当前类加载器
      }
    }
  } // unlocking constantPool

  if (do_resolve) {
    // this_oop must be unlocked during resolve_or_fail
    oop protection_domain = this_oop->pool_holder()->protection_domain();
    Handle h_prot (THREAD, protection_domain);
    // 执行加载。 用当前类加载器尝试去加载目标类
    Klass* k_oop = SystemDictionary::resolve_or_fail(name, loader, h_prot, true, THREAD);
    KlassHandle k;
    if (!HAS_PENDING_EXCEPTION) {
      k = KlassHandle(THREAD, k_oop);
      // preserve the resolved klass.
      mirror_handle = Handle(THREAD, k_oop->java_mirror());
      // Do access check for klasses
      verify_constant_pool_resolve(this_oop, k, THREAD);
    }
}

上面源码，可以证明，A中用到B，会用A的加载器AL去尝试加载B。

另外，如果AL不是bootstrap loader，那么会触发Java层对应对象ClassLoader的loadClass方法，如下：

在loadClass中，一般会选择自己加载，调用defineClass返回Class给虚拟机；或者选择委派，委派给别的加载器去loadClass。

那么，这里，大家可能会想到一个问题，AppClassLoader去加载的java/lang/Float,但这个类最终必然被委派给BootstrapClassLoader去加载，那么下次AppClassLoader还会去再尝试加载一遍java/langFloat吗？

答案是否定的。AppClassLoader尝试加载过Float，即使，它最后不是自己define的，但它也会被记在自己的加载器上，下次JVM层这个加载器需要加载Float这个类，会发现自己已经loadClass过了，则直接返回实际被bootstrapClassLoader define的类Float。

以下图，说明了这种情况：

上图可以看到，一个类加载器L只要尝试过去load一个类A，那么不管最后这个类A的加载器是不是L，这个类A都会被记录在加载器L中，L下次不会再触发A的加载，而是直接从缓存取。

这个在JVM中数据结构的实现，是一个开散列哈希表。ClassLoader和类的名称，计算出hash值来存储。因此ClassLoader+类名唯一确定了一个类。如下图。

JDK父委派机制

public class ClassLoader{
  protected Class<?> loadClass(String name, boolean resolve)
        throws ClassNotFoundException
    {
        synchronized (getClassLoadingLock(name)) {
            // First, check if the class has already been loaded
            Class<?> c = findLoadedClass(name);
            if (c == null) {
                long t0 = System.nanoTime();
                try {
                    if (parent != null) {
                        c = parent.loadClass(name, false);
                    } else {
                        c = findBootstrapClassOrNull(name);
                    }
                } catch (ClassNotFoundException e) {
                    // ClassNotFoundException thrown if class not found
                    // from the non-null parent class loader
                }

                if (c == null) {
                    // If still not found, then invoke findClass in order
                    // to find the class.
                    long t1 = System.nanoTime();
                    c = findClass(name);

                    // this is the defining class loader; record the stats
                    sun.misc.PerfCounter.getParentDelegationTime().addTime(t1 - t0);
                    sun.misc.PerfCounter.getFindClassTime().addElapsedTimeFrom(t1);
                    sun.misc.PerfCounter.getFindClasses().increment();
                }
            }
            if (resolve) {
                resolveClass(c);
            }
            return c;
        }
    }
}

可以看出来，当一个ClassLoader被要求加载某个类时，它会优先委派给父加载器去loadClass，当父加载器加载到了，则自己将其return。如果所有父加载器都加载不到，则自己去findClass，再由自己定义。 否则，抛ClassNotFoundException。

那么结合上面说的JVM的一些知识点，这个要加载的类最终会挂载哪些加载器上？ 1. 实际去define它的loader 2.第一个被JVM调用loadClass的加载器。 （就像前面的java/lang/Float会挂在BootstrapClassLoader和AppClassLoader）

目标类代码植入

例如，有一个类：

public class Test{
  public int add(int a, int b){
    return a + b;
  }
}

那么，为了控制这个类的行为，sandbox的transformer会往目标类（EventWatchBuilder控制）中，插入一些代码，插入后的add方法长这样：

public class Test{
  public int add(int a, int b){
    try{
      Object[] params = new Object[]{a, b};
      Spy.Ret retOnBefore = Spy.spyMethodOnBefore(params, 10001); // 10001是指：用户的Event的编号，会用Map绑定
      if(retOnBefore.state == I_RETURN) return (int) retOnBefore.object;
      if(retOnBefore.state == I_THROWS) throws (Throwable) retOnBefore.object;

      a = params[0]; b = params[1];
      int r = a + b;

      Spy.Ret retOnReturn = Spy.spyMethodOnReturn(r, 10001);
      if(retOnReturn.state == I_RETURN) return (int) retOnReturn.object;
      if(retOnReturn.state == I_THROWS) throws (Throwable) retOnReturn.object;

      return r;
    }catch (Throwable cause){
      Spy.Ret retOnThrows = Spy.spyMethodOnThrows(cause, 10001);
      if(retOnThrows.state == I_RETURN) return (int) retOnThrows.object;
      if(retOnThrows.state == I_THROWS) throws (Throwable) retOnThrows.object;
      throws cause;
    }
  }
}

Spy方法我省去了一些非核心的参数，便于观看

可以看到，往add方法的 方法开始时，return前，throws前，埋藏了改变程序执行逻辑的程序段。例如before（函数开始时），根据Spy.onBofore返回值retOnBefore的state，会选择是直接return(I_RETURN)还是抛异常(I_THROWS)还是不影响程序执行(I_其他)。

那么Spy类是什么呢？这个Spy类在sandbox-spy模块。前面拿到了Instrumentation，可以通过appendToBootstrapClassLoaderSearch，将Spy类所在的jar包，加入到BootstrapClassLoader的扫描路径，只要Spy被挂在根加载器，那么所有的Java类都能够访问到这个类。

public class Spy{
  public static Ret spyMethodOnBefore(final Object[] argumentArray,
                                        final int listenerId) throws Throwable {
      try {
          final SpyHandler spyHandler = namespaceSpyHandlerMap.get(namespace);
          if (null == spyHandler) {
              return Ret.RET_NONE;
          }
          return spyHandler.handleOnBefore( // 最后会转发给对应的Event
                  listenerId, argumentArray
          );
      } catch (Throwable cause) {
          handleException(cause);
          return Ret.RET_NONE;
      } finally {
          selfCallBarrier.exit(thread, node);
      }
  }
}

handleOnBefore最终会走到：

public class EventListenerHandler implements SpyHandler {

  private Spy.Ret handleEvent(final int listenerId,
                                  final int processId,
                                  final int invokeId,
                                  final Event event,
                                  final EventProcessor processor) throws Throwable {
    final EventListener listener = processor.listener;

    // 如果当前事件不在事件监听器处理列表中，则直接返回，不处理事件
    if (!contains(processor.eventTypes, event.type)) {
        return newInstanceForNone();
    }
    try {
        listener.onEvent(event); // 触发用户编写的onEvent事件
    }
    // 代码执行流程变更
    catch (ProcessControlException pce) {
        final EventProcessor.Process process = processor.processRef.get();
        final ProcessControlException.State state = pce.getState();
        switch (state) {
            // 立即返回对象
            case RETURN_IMMEDIATELY: {
                // 让流程立即返回
                return Spy.Ret.newInstanceForReturn(pce.getRespond());
            }
            // 立即抛出异常
            case THROWS_IMMEDIATELY: {
                final Throwable throwable = (Throwable) pce.getRespond();
                // 让流程立即抛出
                return Spy.Ret.newInstanceForThrows(throwable);

            }
            // 什么都不操作，立即返回
            case NONE_IMMEDIATELY:
            default: {
                return newInstanceForNone();
            }
        }
    }
    // 默认返回不进行任何流程变更
    return newInstanceForNone();
  } 
}

在用户编写的onEvent中，会利用ProcessController来控制目标程序流，以直接return为例:

public class ProcessController{
  public static void returnImmediately(final Object object) throws ProcessControlException {
      throw new ProcessControlException(RETURN_IMMEDIATELY, object);
  }
}

这里抛出的异常，在EventListenerHandler的handleEvent中被接住，返回对应的对象给Spy类，然后再返回给Test中的add方法，最终达到用户的event控制add方法的行为的目的。

sandbox在植入代码前，都会先插入ICONST_2,POP， 在植入一段代码后，都会插入ICONST_3,POP。

ICONST_2
POP

// sandbox插入的一段指令。。。

ICONST_3
POP

然而，ICONST_X，然后POP。即：压栈，再弹出，对字节码解释执行没有任何影响。那么这个操作是什么用的呢？

我个人认为， 这是用于还原字节码的。如果我想去扒取(例如althas)一个正在运行应用的字节码，那么拿到的其实可能是被sandbox注入过指令的字节码。那么有个这个ICONST_X,POP标记（正常编译器不会生成这组无用指令），就可以在应用外部，轻松清除清除sandbox的注入，拿到原始的字节码，用于后续可能进行的分析。干净又卫生。
